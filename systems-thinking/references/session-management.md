# 会话管理操作手册

本文件定义了系统思考技能的会话持久化机制。所有会话通过文件读写实现跨对话记忆。

## 目录结构

```
~/.systems-thinking/
└── sessions/
    ├── 2026-04-08-team-communication.md
    ├── 2026-04-05-product-strategy.md
    └── 2026-04-01-employee-retention.md
```

- **路径**：`~/.systems-thinking/sessions/`（用户主目录下，跨项目可用）
- **首次使用**：运行 `mkdir -p ~/.systems-thinking/sessions/` 创建目录
- **文件命名**：`YYYY-MM-DD-{问题关键词}.md`
- **关键词**：从核心问题提取 2-3 个词，仅使用小写字母、数字和连字符

## 会话文件格式

每个会话文件由 YAML frontmatter 和 Markdown 正文组成：

```markdown
---
id: "2026-04-08-team-communication"
created: "2026-04-08T14:30:00"
updated: "2026-04-08T15:45:00"
stage: 3
status: "in-progress"
problem_summary: "团队沟通效率低下，跨部门协作经常出问题"
---

# 系统思考分析：团队沟通效率低下

## 阶段一：问题边界 ✅

### 核心问题
[一句话描述]

### 关键因素
1. [因素1]
2. [因素2]
...

### 最关键的驱动因素
[描述]

### 系统边界
- **时间范围**：[短期/中期/长期]
- **空间范围**：[组织/部门/团队]
- **利益相关者**：[列表]

### 理想结果
[描述]

## 阶段二：因果关系 ✅

### 变量清单
| 变量名 | 当前趋势 | 备注 |
|--------|----------|------|
|        |          |      |

### 因果链
[变量A] --S--> [变量B]  (说明)

## 进度备注
- 当前阶段：[描述]
- 下次继续：[具体要点]

```

### Frontmatter 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 会话唯一标识，格式 `YYYY-MM-DD-{slug}` |
| `created` | ISO datetime | 创建时间 |
| `updated` | ISO datetime | 最后更新时间 |
| `stage` | number | 当前阶段（1-5） |
| `status` | string | `in-progress` / `report-ready` / `completed` |
| `problem_summary` | string | 问题一句话摘要，用于列表展示 |

### 阶段状态标记

- `✅` 已完成
- `🔄` 进行中
- `⬜` 未开始

## 启动流程

### 步骤一：检查会话目录

使用 Bash 工具执行：
```bash
ls -t ~/.systems-thinking/sessions/ 2>/dev/null
```

### 步骤二：根据结果分流

**情况 A：无会话或目录不存在**
- 执行 `mkdir -p ~/.systems-thinking/sessions/`
- 直接进入新会话流程

**情况 B：有历史会话**
- 读取每个文件的 frontmatter（只需读取前 10 行获取 YAML 头）
- 按更新时间倒序排列
- 展示会话选择菜单

### 会话选择菜单格式

```
📂 发现历史分析会话：

  1. [团队沟通效率低下] — 阶段 3/5（回路识别中）— 4月8日
  2. [产品定价策略优化] — 阶段 5/5（待确认报告）— 4月5日
  3. [员工流失问题分析] — ✅ 已完成 — 4月1日

──────────────────────────────
请选择：
• 输入数字继续对应会话
• 输入 n 开始新的分析
```

### 步骤三：处理用户选择

**选择继续会话（输入数字）：**
1. 完整读取选中的会话文件
2. 展示恢复摘要（见下文）
3. 等用户确认后从当前阶段继续

**选择新建会话（输入 n）：**
1. 进入阶段一采访
2. 完成阶段一后创建会话文件

## 恢复会话摘要格式

继续会话时，向用户展示以下摘要：

```
🔄 欢迎回来！

上次我们正在分析【{problem_summary}】
已完成到阶段 {stage}/5。

✅ 已完成：
  • 阶段一：问题边界 — {一句话概括}
  • 阶段二：因果关系 — 识别了 {N} 个变量和 {M} 条因果链
  • ...

接下来我们将进入阶段 {stage+1}：{阶段名称}。

直接继续？还是想回顾或修改之前的发现？
```

如果用户选择回顾，逐阶段展示发现并允许修改。

## 保存检查点

### 阶段完成时保存

每完成一个阶段后，**必须**执行以下操作：

1. 使用 Write 工具更新会话文件
2. 更新 frontmatter 中的 `stage` 和 `updated` 字段
3. 填充对应阶段的发现内容
4. 在"进度备注"中记录当前状态和下次继续的要点
5. 告知用户："✅ 阶段 {n} 完成！进度已保存。"

### 用户要求暂停时保存

当用户说"先到这里"、"暂停"等时：

1. 保存当前所有已确认的发现到会话文件
2. 更新"进度备注"，记录讨论到了哪里、下一步是什么
3. 告知用户："💾 进度已保存。下次使用 `/systems-thinking` 即可继续。"

## 报告前确认流程

阶段 5 完成后，**不要立即生成报告**，而是先展示确认摘要：

```
📋 分析完成，生成报告前请确认：

━━━━━━━━━━━━━━━━━━━━━━━━━━
核心问题：{问题}
关键变量：{N} 个
反馈回路：{X} 个增强回路 + {Y} 个调节回路
━━━━━━━━━━━━━━━━━━━━━━━━━━

🔑 关键杠杆解：
  1. {杠杆解1}
  2. {杠杆解2}

⚡ 推荐行动：
  • 短期：{行动1}
  • 中期：{行动2}

以上内容是否准确完整？
• 确认 → 我将生成完整分析报告
• 修改 → 告诉我哪里需要调整
• 补充 → 我可以继续深入某个部分
```

用户确认后：
1. 生成完整报告（使用 `references/output-template.md` 模板）
2. 更新会话文件 `status` 为 `completed`
3. 告知用户报告已生成

## 会话删除

用户可以要求删除历史会话：

```
请确认要删除以下会话？
  [{problem_summary}] — {date}

删除后无法恢复。确认删除？(y/n)
```

确认后使用 Bash 删除文件。

## 边界情况处理

| 情况 | 处理方式 |
|------|----------|
| 会话文件为空或损坏 | 视为无效会话，从列表中排除，提示用户 |
| frontmatter 缺少字段 | 使用默认值：stage=1, status="in-progress" |
| 目录不存在 | 自动创建 `mkdir -p` |
| 用户在不同项目调用 | 会话在用户主目录，与项目无关 |
| 阶段内容为空但有阶段标记 | 尊重标记，但提示用户可能需要重新讨论该阶段 |
| 同一天创建多个会话 | 文件名使用不同 slug 区分 |

## 会话文件名生成规则

从用户描述的核心问题生成文件名 slug：

1. 中文：提取 2-4 个关键词，用拼音或英文翻译
2. 英文：提取 2-3 个核心单词
3. 全部小写，空格替换为连字符
4. 移除特殊字符，只保留 `[a-z0-9-]`
5. 长度限制：slug 部分不超过 40 个字符

示例：
- "团队沟通效率低下" → `2026-04-08-team-communication.md`
- "产品功能蔓延问题" → `2026-04-08-feature-creep.md`
- "为什么员工流失率居高不下" → `2026-04-08-employee-turnover.md`
