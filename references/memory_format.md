# 议题追踪格式规范

## 议题分类：

**议题层次说明**：使用 Step 4a 加载的议题层次定义。

## 议题记录格式

```markdown
#### [议题名称]
- **首次提及**: YYYY-MM-DD
- **最后更新**: YYYY-MM-DD
- **当前进度**: （根据议题特点，描述讨论到什么程度了）
- **关键结论**:
  - [日期] **背景**：为什么讨论这个问题 → **讨论**：主要观点和争论 → **结论**：最终达成什么共识或发现
- **待解决**: 问题列表
```

## 结论记录规范
遵循 Step 4a 加载的结论写作规范。每个结论必须包含完整的因果链（背景→讨论→结论）。

## 工程待解决问题

在记忆文档末尾（金句记录之前）维护此子节，**仅记录工程报告中状态为 ⚠️ 或 🔄 的条目**（📝 不写入）。

```markdown
## 工程待解决问题

| 首次记录 | 分组 | 问题描述 | 当前状态 | 状态描述 |
| --- | --- | --- | --- | --- |
| 2026-03-19 | 记忆与进化机制 | 上下文丢失，三段式方案未落地 | 🔄 | 方案待验证 |
| 2026-03-19 | 部署与基础设施 | 定时任务稳定性差，从未成功触发 | ⚠️ | 待解决 |
```

**更新规则**：

- 新增：每次日报后将新出现的 ⚠️/🔄 条目追加进来

- 解决：问题在后续工程报告中变为 ✅ 时，从此表删除对应行

## 记忆更新规则

每次生成日报后：

1. 更新相关议题的"最后更新"时间

2. 添加新的关键结论（必须包含背景→讨论→结论）

3. 更新进度描述

4. 记录待解决问题

5. 同步更新 `## 工程待解决问题` 子节（仅 ⚠️/🔄，不含 📝）

## Token管理策略

| 策略   | 触发条件     | 执行动作               |
| ---- | -------- | ------------------ |
| 结论压缩 | 单议题>800字 | 保留最近5条完整结论，其余压缩为摘要 |
| 议题归档 | 7天无更新    | 移至归档区，保留摘要         |
| 月度清理 | 跨月时      | 将上月归档移至独立文件        |

**Token预算**: 总计 < 10000 tokens

使用 Step 4a 加载的成员角色定义。具体人名和角色从 Step 1a 配置获取。
